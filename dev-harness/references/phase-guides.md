# Phase Guides — 외부 스킬 Fallback Inline 가이드

`config.yaml`의 외부 스킬 호출이 실패하거나 사용자가 `type: inline`을 선택한 경우, Claude가 따라가는 최소 가이드라인.

---

## #brainstorm

외부 스킬 미사용 시 brainstorming 진행 룰:

1. **목적 확인** — 사용자 요청의 본질 한 줄로 재진술
2. **제약 수집** — 기술 스택, 기존 코드 패턴, 외부 의존성 명시
3. **성공 기준** — 무엇이 충족되면 완료인지 측정 가능한 형태로 (예: "X 함수가 Y 인풋에 Z 리턴")
4. **대안 2-3개** — 접근법 차이를 트레이드오프와 함께 제시
5. **사용자 승인** — 선택된 디자인을 한 단락으로 정리, 사용자 OK 받기

⚠️ 추측·확장 금지. 사용자가 명시하지 않은 요구사항은 한 줄 질문 후 진행.

---

## #plan

외부 스킬 미사용 시 plan 작성 룰:

1. **파일별 변경 목록** — 추가/수정/삭제할 파일 경로 + 한 줄 변경 요약
2. **순서 의존성** — 어느 변경이 먼저 끝나야 다음이 가능한지 명시
3. **TDD 단계** (가능 시):
   - Red: 실패하는 테스트 작성
   - Green: 최소 코드로 통과
   - Refactor: 정리
4. **검증 명령** — 각 단계 종료 시 실행할 명령 (예: `npm test`, `pytest path/to/file.py`)
5. **롤백 지점** — git tag `safe/exec-<ts>`로 마킹할 시점 결정

출력 형식: 마크다운 체크리스트 (각 항목 `- [ ]`).

---

## #execute

외부 스킬 미사용 시 implementation 진행 룰:

1. **plan의 첫 미완 항목 진행** — 순서대로
2. **단일 책임 단계** — 한 단계에서 한 파일/한 함수만 변경
3. **검증 즉시 실행** — 단계 종료 시 plan에 명시된 검증 명령 실행
4. **실패 시 정지** — 검증 실패하면 다음 단계로 넘어가지 말고 사용자에게 보고
5. **commit per 단계** — 의미 있는 단위마다 git commit (`feat:` `fix:` `refactor:` prefix)

⚠️ plan을 무시하고 한꺼번에 끝내려 하지 말 것. 점진적 검증이 핵심.

---

## #qa

외부 스킬 미사용 시 QA 진행 룰:

1. **단위 테스트** — 변경된 모듈의 기존 + 신규 테스트 실행
2. **타입 체크** — `tsc --noEmit`, `mypy`, `go vet` 등 프로젝트 표준
3. **린트** — 프로젝트 표준 린터 (eslint, ruff 등)
4. **수동 시나리오** — plan에 명시된 핵심 시나리오 1개를 실행/관찰
5. **회귀 체크** — 변경되지 않은 모듈 중 의존성 있는 것의 테스트도 1회 실행

판정:
- **PASS**: 위 5개 모두 통과
- **FAIL**: 1개라도 실패 → `blocked_reason` 기록 + 사용자에게 (a) Execute 재진입 / (b) Lesson 기록 후 종료 선택

---

## 공통 룰 (모든 phase)

- **재투입된 lesson 우선** — Phase 0에서 인용된 Rule들이 이 가이드보다 우선. 충돌 시 lesson Rule 따름
- **사용자 답변 대기 시 멈춤** — 한 줄 질문 후 응답 전에 다음 단계로 진행 금지
- **state.json 갱신** — 각 phase 완료 시 `completed_phases`에 phase명 append, `current_phase` 다음 단계로
