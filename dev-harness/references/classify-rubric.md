# Classify Rubric — 복잡도 점수 + Tier 결정

Phase 1 (Classify)에서 Claude가 사용자 요청을 분석하여 복잡도 점수를 계산하고 Tier를 결정하는 룰북.

## 6개 기준 점수표

| 기준 | 점수 | 판정 가이드 |
|------|------|------------|
| 변경 파일 수 | **1점/파일 (max 5)** | 새 파일 + 수정 파일 합계. 5개 초과해도 5점으로 캡 |
| 크로스서비스 의존 | **+3** | 여러 서비스/모듈 간 변경 필요 (예: API + DB + 프론트) |
| 신규 기능 | **+2** | 기존에 없던 기능 신설. 버그 수정/리팩토링은 0점 |
| 아키텍처 결정 | **+3** | 구조적 결정 동반 (새 추상화 도입, 패턴 변경, 라이브러리 선택) |
| 보안/인증/결제 | **+4** | 인증·인가·결제·시크릿·암호화 관련 변경 |
| DB 스키마 변경 | **+3** | 테이블/컬럼 추가·삭제·변경, 마이그레이션 동반 |

합계 = Sum(applicable). 중복 적용 가능 (예: 신규 기능 + 아키텍처 결정 + 보안 = +9).

## Tier 임계치

| 합계 | Tier | MVP 지원 |
|------|------|---------|
| 0–3 | **Tier 1** | ✅ |
| 4–8 | Tier 2 | ❌ |
| 9+ | Tier 3 | ❌ |

## Fail-Loud 게이트 (MVP 핵심 룰)

**자동 다운그레이드 금지.** 점수가 4+로 Tier 1 임계치를 초과하면 다음 중 사용자가 명시 선택해야 한다:

```
complexity=<X> → Tier <N>. MVP는 Tier 1(≤3)만 지원.
(a) 강제 Tier 1으로 진행 (위험: 단순화로 인한 누락 가능)
(b) 중단 + lessons.md에 "범위 초과" entry 기록 후 사용자 재설계 권유
```

침묵 다운그레이드 금지 — 채팅에 위 메시지 그대로 출력 후 사용자 답변 대기.

## 출력 형식 (state.json 반영용)

Classify 종료 시 Claude는 다음 JSON 구조를 채팅에 출력하고 state.json에 반영:

```json
{
  "complexity_score": 3,
  "tier": 1,
  "score_breakdown": {
    "files_changed": 1,
    "cross_service": false,
    "new_feature": true,
    "architecture": false,
    "security": false,
    "db_schema": false
  }
}
```

## 판정 시 주의

- **추측 금지**: 사용자 요청에서 명시되지 않은 측면은 0점. 추정·확장 금지
- **모호 시 사용자 확인**: "DB 변경 동반되나요?" 같은 한 줄 질문 후 점수 확정
- **이미 알려진 lesson 적용**: Phase 0에서 재투입된 lesson에 "X 작업은 항상 보안 가산점" 같은 룰이 있으면 적용
