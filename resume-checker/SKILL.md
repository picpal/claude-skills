# 이력서 검토 스킬

한국어 이력서를 받아 인터뷰 → 분석 → 첨삭 → 최종본 생성까지 수행하는 스킬이다.

## 생성 파일

1. **`이력서_첨삭_리포트.md`** — AS-IS / TO-BE 비교 형식의 상세 첨삭 리포트
2. **`이력서_최종본.md`** — 모든 피드백을 반영한 수정 완료 이력서

## 워크플로우

### Phase 1: 이력서 수신

사용자로부터 이력서를 받는다. 지원하는 형식:

| 형식 | 처리 방법 |
|------|-----------|
| PDF (.pdf) | Read 도구로 읽기 |
| Word (.docx) | Read 도구로 읽기 |
| 마크다운 (.md) | Read 도구로 읽기 |
| 텍스트 (.txt) | Read 도구로 읽기 |
| 붙여넣기 텍스트 | 메시지 본문에서 직접 추출 |

이력서를 받으면 전체 내용을 읽고 다음 Phase로 진행한다.

### Phase 2: 지원 정보 인터뷰

이력서 내용을 파악한 뒤, 사용자에게 다음 정보를 **한 번에** 질문한다:

1. **지원 회사명** — 어느 회사에 지원하는가
2. **지원 포지션** — 어떤 직무/역할에 지원하는가
3. **JD(Job Description)** — 채용 공고가 있으면 공유해달라고 안내 (선택사항)

질문 예시:
```
이력서를 확인했습니다. 검토를 위해 몇 가지 여쭤볼게요.

1. 어느 회사에 지원하시나요?
2. 어떤 포지션(직무)에 지원하시나요?
3. 채용 공고(JD)가 있다면 링크나 내용을 공유해주세요. (선택)
```

사용자 답변을 받으면 다음 Phase로 진행한다.
JD가 제공되면 Phase 3에서 직무 적합성 분석의 기준으로 활용한다.

### Phase 3: 이력서 분석

이력서를 5개 관점에서 분석한다. 각 관점마다 해당하는 참조 문서를 읽고 기준에 따라 평가한다.

#### 3-1. 구조 분석

`references/structure-guide.md`를 참조한다.

- 섹션 구성이 적절한가 (필수 섹션 누락 여부)
- 섹션 순서가 논리적인가
- 각 섹션의 분량이 균형 잡혀 있는가
- 불필요하거나 중복되는 섹션이 없는가

#### 3-2. 문체/표현 분석

`references/writing-style.md`를 참조한다.

- 오탈자, 맞춤법 오류
- 비문, 어색한 문장
- 일관성 없는 어미/경어체 혼용
- 모호하거나 추상적인 표현 (예: "다양한 경험", "열심히")
- 성과를 정량적으로 표현했는가

#### 3-3. 직무 적합성 분석

`references/job-fit.md`를 참조한다.

- 지원 포지션에 맞는 핵심 역량이 드러나는가
- JD가 있다면: JD 요구사항 대비 이력서 커버리지 분석
- JD가 없다면: 해당 직무의 일반적 요구사항 기준으로 분석
- 직무와 무관한 내용이 과도하게 포함되어 있지 않은가

#### 3-4. 임팩트/성과 표현 분석

`references/impact-expression.md`를 참조한다.

- 경험이 단순 나열인가, 성과 중심인가
- STAR(Situation-Task-Action-Result) 프레임이 적용되었는가
- 정량적 수치가 포함되어 있는가
- 기여도가 명확히 드러나는가

#### 3-5. 가독성/포맷 분석

`references/readability.md`를 참조한다.

- 글머리 기호, 들여쓰기 일관성
- 날짜/기간 표기 통일성
- 적절한 길이 (경력 연차 대비)
- 핵심 키워드가 눈에 잘 들어오는가

#### 3-6. 강의 기반 종합 분석

강의 참조 문서를 활용하여 다음 관점에서 추가 분석한다.
각 관점마다 해당하는 참조 문서를 읽고 기준에 따라 평가한다.

- **검토자 관점**: `references/lecture-reviewer-perspective.md` 참조
  - 첫인상 설계가 되어 있는가 (N초 규칙)
  - 검토자가 멈출 키워드 앵커가 있는가
  - 검토자 피로를 유발하는 요소가 없는가

- **스토리텔링**: `references/lecture-storytelling.md` 참조
  - 이력 나열이 아닌 Why(배경) → 행동 → 결과 스토리가 있는가
  - 패턴에 갇힌 서술이 없는가

- **흔한 실수 체크**: `references/lecture-common-mistakes.md` 참조
  - 강의에서 경고하는 대표적 실수에 해당하는 항목이 없는가

- **전략적 구성**: `references/lecture-strategy.md` 참조
  - 지원 회사에 맞춤화되어 있는가
  - 면접에서 받고 싶은 질문을 유도하는 구성인가

- **본질 점검**: `references/lecture-essence.md` 참조
  - 이력서가 "이 사람과 일하고 싶다"는 매력을 전달하는가

### Phase 4: 첨삭 리포트 생성

분석 결과를 AS-IS / TO-BE 비교 형식으로 정리하여 첨삭 리포트를 생성한다.

#### 리포트 구조

```markdown
# 이력서 첨삭 리포트

**지원 회사**: [회사명]
**지원 포지션**: [포지션]
**검토일**: [날짜]

---

## 우선순위 요약

반드시 수정해야 할 상위 5개 항목을 중요도 순으로 정리한다.

| 순위 | 항목 | 카테고리 | 영향도 |
|------|------|----------|--------|
| 1 | [항목] | [카테고리] | 높음 |
| 2 | [항목] | [카테고리] | 높음 |
| ... | ... | ... | ... |
```

각 첨삭 포인트는 반드시 AS-IS(원문) → 문제점 → TO-BE(수정안) 형식을 따른다.
원문을 정확히 인용해야 사용자가 어떤 부분을 말하는지 즉시 파악할 수 있다.

#### 첨삭 리포트 작성 시 출력 규칙

- **참조 문서의 내부 출처(강의 번호, 강 번호 등)를 첨삭 본문에 노출하지 않는다.**
  - 금지 예시: "(15강)", "강의 원칙(12강)", "24강 기준", "35강에서는~"
  - 참조 문서는 분석의 판단 근거로만 활용하고, 리포트에는 근거의 내용만 자연스럽게 녹여서 서술한다.
- `📚 참조` 행에는 참조 문서 파일명과 해당 기준만 기재한다.
  - 허용 예시: `lecture-strategy.md` — 자기소개 3줄 제한 기준
  - 금지 예시: `lecture-strategy.md` — 자기소개는 3줄로 끝내라(24강)

### Phase 5: 최종본 생성

첨삭 리포트의 모든 TO-BE 제안을 반영하여 완성된 이력서를 마크다운으로 생성한다.

최종본 작성 규칙:
- 원본 이력서의 전체 구조를 유지하되, 분석에서 제안한 구조 변경이 있으면 반영
- 모든 오탈자와 문체 수정 사항 반영
- 성과/임팩트 표현 개선 반영
- 상단에 메타 정보 표기:

```markdown
# [이름] — 이력서

> 이 문서는 이력서 검토 스킬에 의해 수정된 최종본입니다.
> 원본 대비 변경 사항은 첨삭 리포트를 참고하세요.
> 수정일: [날짜]

---

[수정된 이력서 전문]
```

### Phase 6: 전달

리포트 파일은 사용자의 **바탕화면(Desktop)**에 생성한다.

저장 경로: `~/Desktop/`

생성 파일:
1. `이력서_첨삭_리포트.md` — 상세 첨삭 내용
2. `이력서_최종본.md` — 수정 완료된 이력서

파일명에 사용자 이름이나 회사명을 포함할 수 있다면 포함한다.
예: `~/Desktop/홍길동_네이버_첨삭리포트.md`, `~/Desktop/홍길동_네이버_최종본.md`

전달 시 요약:
```
검토가 완료되었습니다.

1. 첨삭 리포트: [파일 경로] — AS-IS/TO-BE 비교 형식의 상세 피드백
2. 최종본: [파일 경로] — 모든 수정사항이 반영된 이력서

주요 수정 포인트:
- [상위 3개 핵심 수정사항 요약]

추가 수정이 필요하시면 말씀해주세요.
```
