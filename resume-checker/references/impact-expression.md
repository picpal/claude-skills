# 성과/임팩트 표현 가이드

## STAR 프레임워크

경력 기술의 각 항목은 STAR 구조를 갖추면 가장 효과적이다.

| 요소 | 설명 | 이력서 적용 |
|------|------|-------------|
| **S**ituation | 상황/배경 | 간략하게 (1절) |
| **T**ask | 맡은 역할/과제 | 간략하게 (1절) |
| **A**ction | 구체적 행동 | 상세하게 (핵심) |
| **R**esult | 결과/성과 | 정량적으로 (핵심) |

### 이력서에서의 STAR 적용

이력서는 자소서가 아니므로 STAR를 풀어쓰지 않는다. 불릿 포인트 1~2줄에 압축한다.

**나쁜 예 (단순 나열):**
> - 백엔드 API 개발
> - 데이터베이스 설계
> - 서버 모니터링

**좋은 예 (STAR 압축):**
> - 일 평균 50만 요청을 처리하는 주문 API를 설계/구현하여, 장바구니→결제 전환율 12% 향상에 기여 (Spring Boot, Redis)
> - 레거시 DB 스키마를 정규화하고 인덱스를 최적화하여 주요 쿼리 응답 시간 70% 단축
> - Prometheus + Grafana 기반 모니터링 시스템을 구축하여 장애 감지 시간 30분 → 3분으로 단축

## 정량화 가이드

### 정량화할 수 있는 영역

| 영역 | 측정 지표 예시 |
|------|---------------|
| 성능 | 응답 시간, 처리량, 가용률, 에러율 |
| 비즈니스 | 매출, 전환율, DAU/MAU, 이탈률, NPS |
| 효율 | 처리 시간 단축, 자동화율, 비용 절감 |
| 규모 | 사용자 수, 데이터 크기, 트래픽, 팀 규모 |
| 품질 | 버그 감소율, 테스트 커버리지, 코드 리뷰 통과율 |

### 정량화 표현 패턴

```
[무엇을] [어떻게 해서] [결과 수치]
```

예시:
- "결제 모듈 리팩토링으로 트랜잭션 실패율 5% → 0.3%로 감소"
- "CI/CD 파이프라인 구축으로 배포 주기 월 1회 → 주 3회로 개선"
- "10만 DAU 서비스의 검색 기능을 개선하여 클릭률 15% 향상"

### 정량화가 어려울 때

모든 성과를 수치화할 수 있는 것은 아니다. 그럴 때는 규모나 범위로 맥락을 제공한다.

**정량화 불가 시 대안:**
- "8인 팀에서 기술 리드 역할 수행" (규모)
- "월간 100만 PV 서비스 운영" (규모)
- "사내 최초로 마이크로서비스 아키텍처 도입" (최초/유일)
- "팀 전체 도입을 이끈 코드 리뷰 프로세스 확립" (범위)

## 기여도 명확화

팀 프로젝트에서 본인의 역할을 명확히 구분해야 한다.

| 모호한 표현 | 명확한 표현 |
|-------------|-------------|
| "프로젝트에 참여" | "결제 모듈 백엔드 전담 개발" |
| "팀과 함께 진행" | "5인 팀에서 프론트엔드 리드로서 아키텍처 설계 및 코드 리뷰 담당" |
| "서비스 개발에 기여" | "검색 API 설계/구현 및 Elasticsearch 인덱싱 최적화 담당" |

역할을 나타내는 동사 선택도 중요하다:
- **주도**: 설계, 구축, 주도, 리드, 기획
- **실행**: 구현, 개발, 작성, 배포, 운영
- **개선**: 최적화, 리팩토링, 마이그레이션, 자동화
- **협업**: 조율, 리뷰, 멘토링, 교육
